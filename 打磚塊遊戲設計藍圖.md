## 一、遊戲概念

**一句話定位：** 畫面色彩鮮明、操作簡單的網頁打磚塊遊戲。玩家滑動底部擋板，把球彈起去打掉上方的磚塊、水球與水果，清光整面就過關，共 5 關。

**核心體驗：**
- 操作門檻低 —— 一根手指就能玩。
- 即時成就感 —— 每打掉一個目標都有聲音與動畫回饋。
- 兩種難度 —— 小小孩玩無限重生的輕鬆模式，大一點的孩子挑戰有限愛心模式。

---

## 二、主題包裝：三種可破壞目標

上方排列的不再是單調的磚塊，而是三種混搭的可愛目標：

| 目標 | 外觀 | 行為設計 |
|------|------|----------|
| **🧱 磚塊** | 鮮豔圓角色塊 | 基本目標，打 1 下就破。 |
| **💧 水球** | 半透明水滴/氣球 | 打到會「啪」地濺開（水花動畫），可設計成打 1 下破，破時音效最清脆。 |
| **🍉 水果** | 西瓜、草莓、橘子等 | 較硬，需打 2 下：第 1 下出現裂痕，第 2 下整顆爆開噴汁。 |

> 每一關可以混搭這三種目標，甚至排成圖案（笑臉、星星），讓清除過程更有趣、更療癒。

## 三、核心玩法機制

| 元素 | 說明 |
|------|------|
| **擋板（Paddle）** | 畫面底部，可左右滑動。手機用手指拖曳，電腦用滑鼠或 ← → 鍵。 |
| **球（Ball）** | 依物理規則彈跳，碰到擋板、牆壁、目標都會反彈。 |
| **目標（磚塊/水球/水果）** | 排列在畫面上方，被球打到就破（水果需 2 下）。 |
| **邊界（Walls）** | 上、左、右三面是牆會反彈；下方是「死亡線」，球掉下去算失誤。 |
| **過關條件** | 一關的目標全部清除即過關，進入下一關。 |

### 操作方式
- **手機 / 平板：** 手指在螢幕下方左右滑動移動擋板（主打，最直覺）。
- **電腦：** 滑鼠移動，或鍵盤 ← → 鍵。

## 四、雙遊戲模式（懲罰機制）

開始畫面讓玩家先選模式：

### 🌈 輕鬆模式（無限重生）
- 球掉到死亡線下方時，**不會結束遊戲**，球自動重生回到擋板上方。
- 完全沒有失敗壓力，適合最小的孩子或想單純放鬆地玩。
- 仍會記分（清除目標得分），但不會 Game Over。

### ⭐ 挑戰模式（有限愛心）
- 玩家一開始有 **5 顆愛心 ❤️❤️❤️❤️❤️**。
- 球每掉下去一次扣 1 顆愛心。
- 愛心歸零才結束遊戲，顯示「再試一次」按鈕。
- 給小孩足夠的容錯空間（5 顆），但保留一點挑戰感與成就感。

> 兩種模式共用同一套關卡，差別只在死亡線的處理。實作上用一個變數 `gameMode` 切換即可。

## 五、物理原理（球怎麼彈）

打磚塊的靈魂。給開發者的核心邏輯：

1. **基本運動** —— 球有速度向量 `(vx, vy)`，每幀位置加上速度：`x += vx`、`y += vy`。
2. **碰到牆壁反彈**
   - 撞左右牆：`vx = -vx`。
   - 撞上方牆：`vy = -vy`。
3. **碰到目標反彈** —— 依撞到的是上下面還是左右面，反轉對應方向速度，並讓該目標扣血/消失。
4. **碰到擋板反彈（關鍵手感）** —— 不要只是 `vy = -vy`。要依「球打中擋板的哪個位置」決定反彈角度：
   - 打中**正中央** → 幾乎垂直彈回。
   - 打中**越靠邊** → 彈出角度越斜。
   - 這讓玩家能「控制」球的方向，是好玩的重點。
5. **速度建議** —— 兒童版球速偏慢、好預測；加一個最小垂直速度限制，避免球卡在牆間水平來回彈太久。

> 💡 兒童版貼心設計：球速比一般版慢 20～30%，擋板做寬一點，降低挫折感。

## 六、聲音設計（含靜音按鈕）

| 事件 | 建議音效 |
|------|----------|
| 球碰擋板 / 牆 | 輕快的「咚」 |
| 打破磚塊 | 清脆的「叮」 |
| 打破水球 | 「啪嚓」水花聲 |
| 打破水果 | 「噗滋」噴汁聲 |
| 過關 | 歡呼 / 煙火音樂 |
| 失誤掉球 | 溫和的提示音（不要嚇人） |

### 🔇 靜音按鈕（必做）
- 畫面右上角放一個喇叭圖示按鈕。
- 點一下切換「有聲 🔊 / 靜音 🔇」，方便家長隨時控制。
- 建議用一個 `isMuted` 變數控制，靜音時所有音效不播放；圖示同步切換。
- 進階：把使用者的靜音偏好記在 `localStorage`，下次打開記住設定。

## 七、針對小孩的設計考量

- **大、亮、圓潤的視覺** —— 鮮豔色塊、圓角，避免尖銳或灰暗畫面。
- **正向回饋** —— 打掉目標時有悅耳音效＋小爆裂/水花/星星動畫。
- **弱懲罰** —— 用上面的雙模式設計，不讓小孩因為一直失敗而沮喪。
- **過關慶祝** —— 清光時放煙火、彩帶、可愛角色拍手動畫。
- **道具獎勵（進階，可選）** —— 偶爾從破掉的目標掉落「擋板變長」「多球」「慢速」等好康。
- **聲音可關閉** —— 即上面的靜音按鈕。

採「先甜後難」的曲線，讓小孩快速嘗到成功：

| 關卡 | 目標組成 | 排列 | 難度設計 |
|------|----------|------|----------|

| **第 5 關** | 三種大量混搭 | 排成愛心或星星大圖案 | 最終關，球速最快，過關放最盛大的慶祝動畫。 |

## 十、🚀 GitHub + GitHub Pages 部署規則

這是這個專案的核心發布方式，請務必遵守：

### 部署規則
1. **入口檔案必須是 `index.html`，且放在儲存庫「根目錄」**（或統一放在 `/docs` 資料夾，二選一，但要一致）。GitHub Pages 預設會去找根目錄的 `index.html`。
2. **所有檔案路徑用「相對路徑」**，例如 `assets/images/brick.png`、`js/game.js`，**不要用絕對路徑 `/`**。因為 GitHub Pages 的網址帶有子目錄（`/Magic-fruit-splash-breaker/`），用絕對路徑會抓不到檔案。這點對 PWA 尤其重要 —— `manifest.json` 裡的 `start_url` 和 service worker 的快取路徑也都要用相對路徑或帶上正確子目錄，否則離線功能會失效。
3. **儲存庫設為 Public**（公開），免費版 GitHub Pages 才能發布。
4. **分支設定**：在儲存庫 `Settings → Pages` 中，把來源（Source）設為 `main` 分支的根目錄（`/root`）。
5. **純前端、無後端**：GitHub Pages 只能放靜態網站，所以整個遊戲必須是純 HTML/CSS/JS，不能有伺服器端程式。打磚塊正好完全符合。
6. **PWA 需要 HTTPS**：PWA 的 service worker 只能在 HTTPS 環境運作。好消息是 **GitHub Pages 預設就提供 HTTPS**，所以發布上去後 PWA 功能即可正常使用（本機測試則用 `localhost` 也算安全環境）。

### 發布流程
1. 在 GitHub 建立名為 `Magic-fruit-splash-breaker` 的 Public 儲存庫。
2. 把所有遊戲檔案推上去（push）。
3. 進 `Settings → Pages`，Source 選 `main` / `/root`，按 Save。
4. 等一兩分鐘，遊戲就會上線在 `https://你的帳號.github.io/Magic-fruit-splash-breaker`。
5. 之後每次 push 新版本，網站會自動更新。
6. 用手機 Chrome / Safari 打開網址，瀏覽器選單會出現「加到主畫面」，點下去就變成像 App 一樣的圖示了。

> 💡 因為這個規則，開發時就要養成習慣：**圖片和音效一律用相對路徑引用**，本機測試正常不代表上線正常，建議用相對路徑從一開始就避免踩雷。

## 十一、📱 PWA 模式（漸進式網頁應用程式）

把遊戲做成 PWA，玩家就能把它「加到主畫面」，像真正的 App 一樣全螢幕開啟、甚至離線也能玩，對小孩來說體驗更接近原生遊戲。

### PWA 三大必備元件

1. **`manifest.json`（應用程式設定檔）**
   告訴瀏覽器這是一個可安裝的 App，定義名稱、圖示、顏色、開啟方式。範例欄位：
   ```json
   {
     "name": "Magic Fruit Splash Breaker",
     "short_name": "Fruit Breaker",
     "start_url": "./index.html",
     "display": "fullscreen",
     "orientation": "portrait",
     "background_color": "#fff8e1",
     "theme_color": "#ff7043",
     "icons": [
       { "src": "assets/icons/icon-192.png", "sizes": "192x192", "type": "image/png" },
       { "src": "assets/icons/icon-512.png", "sizes": "512x512", "type": "image/png" }
     ]
   }
   ```
   - `display` 用 `fullscreen` 或 `standalone`，讓遊戲開起來沒有瀏覽器網址列。
   - `orientation` 可鎖定 `portrait`（直式）或 `landscape`（橫式），依你的遊戲畫面決定。
   - `start_url` 用相對路徑，配合 GitHub Pages 子目錄。

2. **`service-worker.js`（服務工作者）**
   負責把遊戲檔案快取起來，讓玩家**離線也能玩**。基本任務：
   - 安裝時：把 `index.html`、CSS、JS、圖片、音效全部快取下來。
   - 之後載入：優先從快取讀取（速度快、可離線），快取沒有才上網抓。

3. **App 圖示（Icons）**
   準備至少 `192x192` 和 `512x512` 兩種尺寸的 PNG 圖示，放在 `assets/icons/`。建議用一顆可愛的水果或水球當圖示，符合主題。可用 Kenney 的素材或免費工具自己做。

### 在 `index.html` 裡要做的事
- 在 `<head>` 加一行連結 manifest：
  ```html
  <link rel="manifest" href="./manifest.json">
  <meta name="theme-color" content="#ff7043">
  ```
- 在 JS 裡註冊 service worker（記得用相對路徑）：
  ```javascript
  if ('serviceWorker' in navigator) {
    navigator.serviceWorker.register('./service-worker.js');
  }
  ```

### PWA 注意事項
- **必須 HTTPS** —— GitHub Pages 已內建，發布後即可用；本機測試用 `localhost` 也算安全環境。
- **離線快取要記得更新** —— 改版後若玩家還看到舊版，通常是 service worker 的快取沒更新。慣例做法是在 service worker 裡放一個「快取版本號」，每次改版就改號碼，強制更新。
- **可安裝性檢查** —— 用 Chrome 開發者工具的 **Lighthouse** 跑一次，會告訴你 PWA 還缺什麼（manifest、service worker、圖示是否齊全）。

## 十二、開發步驟（Roadmap）

1. **原型** —— 用色塊做出能動的擋板、會彈的球、一排會消失的目標。只驗證物理手感，不放美術。
2. **核心玩法** —— 加入三種目標（磚塊/水球/水果）、過關判定、計分。
3. **雙模式** —— 加入開始畫面與模式選擇（無限重生 / 5 顆愛心）。
4. **5 關卡** —— 依上面的表格設計 5 關的目標排列。
5. **美術與音效** —— 換上正式圖片與音效（來源見下一節），加靜音按鈕。
6. **兒童體驗優化** —— 調整球速、擋板寬度、加入慶祝動畫。
7. **PWA 化** —— 加入 `manifest.json`、`service-worker.js`、App 圖示，讓遊戲可安裝、可離線。
8. **部署** —— 依第十節推上 GitHub、開啟 Pages，用手機測試「加到主畫面」。
9. **測試** —— 找真實小孩試玩，觀察卡關處再微調。

## 十四、⚠️ 授權注意事項（很重要！）

免費 ≠ 可以隨便用。下載前一定要看清楚每個檔案的授權條款：

- **CC0（公眾領域）** —— 最自由。可商用、可修改、**不需標註作者**。Kenney 大多是這種，最省心。
- **CC-BY（姓名標示）** —— 可免費使用（含商業），但**必須標註原作者**（放在遊戲的「製作群 / Credits」畫面即可）。
- **CC-BY-NC（非商業）** —— 只能用在**不收費**的專案；若日後想賣這遊戲就不能用。

> **安全建議：** 優先全部使用 **Kenney（CC0）** 的素材，最不會有授權問題。若用到 CC-BY 素材，記得在遊戲裡加一頁致謝名單。GitHub 公開儲存庫的 `README.md` 裡也建議列出素材來源與授權，養成好習慣。
